\section{Original Project Vision}

\taustart{T}he original vision for this project was to create an event-driven analytics platform that makes use of serverless technologies in order to efficiently ingest, process and store data. The platform was intended to be scalable, flexible and resource-efficient by leveraging cloud services.

The basic pipeline consists of a client application that generates events and sends them to a serverless ingestion layer in the cloud. This ingestion layer is responsible for receiving the events and passing them to an event stream. From there, downstream processing functions can consume the events, perform analytics and store the results in a database.


\section{The client application}

The client application is a multiplatform mobile app developed using Kotlin Multiplatform Mobile (KMM). Its main purpose is to allow people who engage in Bouldering to log their climbing sessions and track the boulders they have climbed. The logging of climbing sessions thereby generates events that can be sent to the serverless ingestion layer. These events should be handled asynchronously in order to not block the user interface of the app and to sync data after periods of offline usage. \section{Choosing the cloud provider and services}

Choosing the right cloud provider and services for the prototype was a crucial step in the project. After evaluating various options, Google Cloud Platform (GCP) was selected, in order to experiment with Cloud Functions and Pub/Sub for the serverless ingestion layer and event streaming.

However, due to problems with our GCP account, we had to change plans and decided to use a private cloud based on OKD (OpenShift Kubernetes Distribution) for the implementation of the serverless ingestion layer and event streaming. This change would allow us to continue with the project while still exploring serverless and event-driven architectures, as well as gaining experience with Kubernetes-based solutions, which carry less vendor lock-in compared to managed cloud services.

However, this change also meant that we had to adapt our original vision and implementation plan to fit the capabilities and limitations of the private cloud environment. The architecture of the prototype is shown in Figure \ref{fig:architecture}.

\subsection{IaC provisioning - OpenTofu}

For the infrastructure provisioning of the private cloud environment, we decided to use OpenTofu, an Infrastructure as Code (IaC) tool, which is an open-source variant of Terraform. With OpenTofu, we were able to define and manage the infrastructure resources required for our serverless ingestion layer and event streaming in a declarative manner. The explicit goal was to follow a cloud-first approach, where the infrastructure is defined and managed through code, in order to ensure consistency and reproducibility.


\subsection{Serverless ingestion layer - Knative}

The serverless ingestion layer was implemented using Knative, a Kubernetes-based platform specifically designed for building and deploying serverless applications. Knative is also an open-source project, which aligns well with our choice of using a private cloud based on OKD. It provides an operator that should make it easy to install on the cluster and offers features such as automatic scaling, eventing and routing.

\subsection{Event streaming - Apache Kafka}

Kafka is a distributed event streaming platform that is widely used for building real-time data pipelines and streaming applications. It is a technology that we wanted to gain experience with, as it is a popular choice for event-driven architectures.

In hindsight, Kafka might have been an overengineered choice for our prototype, as our event streaming requirements were relatively simple and could have been fulfilled with simpler solutions. However, using Kafka provided us with valuable experience in working with a complex event streaming platform and understanding its capabilities and limitations.

\subsection{NoSQL data storage - MongoDB}

MongoDB was chosen as the data store for storing the climbing session data. The document-style storage model is perfect for the climbing session data, which can be represented as documents. MongoDB also offers scalability and schema flexibility, which are important for our use case.


\section{Implementation}

The implementation of the prototype was done in several steps, starting with the preparation of the cluster environment, followed by the development of the serverless ingestion layer and the event streaming components. The client application was developed in parallel.

\subsection{Preparing the cluster}

The first step was to set up the OKD cluster and install the necessary components. Knative was installed using the Knative operator, which simplified the installation process, but has to be done by the cluster administrator and therefore is not able to be provisioned automatically. Since using the Knative operator needs additional cluster roles, we had to wait for the cluster administrator. In order to start working on the implementation, we used a local Minikube cluster in the meantime. The operator can also be used to install Knative on Minikube.

The operator is installed by running the following command:

\begin{lstlisting}[language=bash, caption={Install Knative operator}]
# install the Knative operator    
kubectl apply -f https://github.com/knative/operator/releases/download/knative-v1.20.1/operator.yaml

# ensure the correct namespace is set
kubectl config set-context --current --namespace=knative-operator

# verify the installation
kubectl get deployment knative-operator
\end{lstlisting}

For the networking layer, we chose the Kourier Ingress, which is installed by creating a KnativeServing custom resource for Knative Serving:

\begin{lstlisting}[caption={Knative Serving with Kourier Ingress}]
apiVersion: operator.knative.dev/v1beta1
kind: KnativeServing
metadata:
  name: knative-serving
  namespace: knative-serving
spec:
  ingress:
    kourier:
      enabled: true
  config:
    network:
      ingress-class: "kourier.ingress.networking.knative.dev"
\end{lstlisting}

Then, Eventing can also be installed using the operator by creating the respective custom resources:

\begin{lstlisting}[caption={Knative Eventing resource}]
apiVersion: v1
kind: Namespace
metadata:
  name: knative-eventing
---
apiVersion: operator.knative.dev/v1beta1
kind: KnativeEventing
metadata:
  name: knative-eventing
  namespace: knative-eventing
\end{lstlisting}

While the Knative operator simplifies the installation process, we encountered some challenges during the installation, where the activator pod was not starting correctly due to not reaching the Knative Autoscaler service. A lot of time was spent debugging this issue, which could not be resolved despite our efforts.

\subsection{Kafka}

The Kafka deployment was provisioned using OpenTofu, defining all infrastructure resources declaratively. The implementation uses Apache Kafka version 4.1.1 running in KRaft mode (Kafka Raft), which eliminates the dependency on Apache Zookeeper using Kafka's built-in consensus protocol. The deployment runs a single node that serves both as a broker and controller. A persistent volume claim is used to ensure data persistence across pod restarts, which is also provisioned via OpenTofu.

The infrastructure definition includes three main Kubernetes resources:

\begin{enumerate}
    \item A \textbf{Deployment} that manages the Kafka pod lifecycle and environment variables for KRaft configuration
    \item A \textbf{Service} that provides stable network endpoints for both broker and controller
    \item A \textbf{Job} that can automatically create the necessary topics after Kafka becomes available
\end{enumerate}

This workflow allows to easily set up the Kafka infrastructure with a single command.

\subsection{MongoDB}

MongoDB was chosen because the climbing session data is naturally represented as documents, making MongoDB's document-based storage model a perfect fit. In most cases, climbing sessions are loaded with all their associated data, such as boulder attempts and location. In the app this is represented as relational data, which requires multiple joins, making queries more complex and less efficient.

The MongoDB deployment was also done completely using OpenTofu with the Bitnami MongoDB Helm chart, which provides a production-ready MongoDB installation with minimal configuration. The data is persisted in a PersistentVolumeClaim to ensure the data survives pod restarts. The application database (\texttt{justgoup}) is automatically created during deployment.

For development purposes, an automated data seeding mechanism was also implemented as a Kubernetes Job. This job executes after MongoDB deployment inserts sample data with indexes for query optimization.

Additionally, Mongo Express was deployed to provide a web-based interface for interacting with the MongoDB instance during development.

\section{Results}

At the end of the project, we were able to implement parts of the original vision, but not the complete connected system. The client application was developed and is able to log climbing sessions, but the integration with the serverless ingestion layer and event streaming components is not fully functional yet. While all parts of the system were implemented individually, the complete end-to-end data flow from the client application to the database is not yet established.

\subsection{Automated Infrastructure Provisioning}

All infrastructure components are defined declaratively using OpenTofu. Using OpenTofu ensures that the infrastructure can be recreated reliably on any machine with a sufficiently prepared cluster, manual setup steps are minimized as much as possible, configuration changes are version-controlled, and the local development environment resembles the later OKD deployment as closely as possible.

The infrastructure can be provisioned or removed using a single command:

\begin{verbatim}
tofu apply
tofu destroy
\end{verbatim}

This approach significantly reduces setup errors and improves reproducibility.

\subsection{Scalability}

Scalability is considered at both the application and infrastructure level.

\subsubsection*{Infrastructure Perspective}

From an infrastructure perspective:

\begin{itemize}
    \item All services are deployed as Kubernetes workloads and can be scaled horizontally.
    \item Kafka enables decoupled, asynchronous communication between producers and consumers. New consumers can be added without modifying existing services.
\end{itemize}

\subsubsection*{Architectural Perspective}

From an architectural perspective:

\begin{itemize}
    \item Event-based communication via Kafka avoids tight coupling.
    \item MongoDB supports horizontal scaling and flexible data models.
    \item Stateless services (e.g., ingestion services) can be replicated easily.
\end{itemize}

While the local Minikube setup uses single replicas for simplicity, the architecture is designed so that scaling up in a real cluster (e.g., OKD) only requires configuration changes, not architectural changes.

\subsection{Fault Tolerance}

Fault tolerance is addressed through both platform mechanisms and architectural decisions.

\subsubsection*{Kubernetes Mechanisms}

Kubernetes provides:

\begin{itemize}
    \item Automatic pod restarts on failure
    \item Health monitoring
    \item Rolling updates
\end{itemize}

\subsubsection*{Messaging and Data Persistence}

Kafka contributes to fault tolerance by:

\begin{itemize}
    \item Persisting events independently of consumers
    \item Allowing consumers to recover and reprocess messages
    \item Decoupling producers from downstream systems
\end{itemize}

MongoDB persistence is handled via PersistentVolumeClaims, ensuring that:

\begin{itemize}
    \item Data survives pod restarts
    \item Temporary failures do not result in data loss
\end{itemize}

This combination ensures that failures do not break the overall system. Of course this also depends on the specific failure scenario and can be further improved in a production environment.

\subsection{NoSQL}

Compared to a relational database, MongoDB reduces complexity for storing event data and enables flexible experimentation during development.

\subsection{Replication}

Replication is used implicitly through the chosen technologies and can be extended further in production environments.

In the current setup:

\begin{itemize}
    \item Kafka supports topic replication (configured minimally for local development)
    \item MongoDB is deployed as a standalone instance locally but supports replica sets in production
    \item Kubernetes allows multiple replicas of stateless services
\end{itemize}

Although the local Minikube setup uses single-node configurations for simplicity, the architecture is replication-ready and can be extended without redesign.

\subsection{Costs}

The solution is designed to minimize costs during development while remaining cloud-ready.

For local development:

\begin{itemize}
    \item Minikube runs entirely on the developer's machine
    \item No cloud resources are consumed
    \item Costs are effectively zero
\end{itemize}

In a cloud or OKD environment:

\begin{itemize}
    \item Costs scale with actual resource usage
    \item Services can be scaled independently
    \item Managed Kubernetes platforms reduce operational overhead
\end{itemize}

Compared to a traditional non-cloud solution:

\begin{itemize}
    \item No dedicated hardware is required
    \item Scaling does not require upfront investment
    \item Infrastructure can be adjusted dynamically
\end{itemize}

\section{Lessons learned}

\subsection{Working with OpenTofu}

OpenTofu proved to be a powerful tool for managing infrastructure as code. It effectively automated the entire process of provisioning and deprovisioning resources and made it easy to track the changes to the infrastructure over time.

Some resources could not be fully managed with OpenTofu, such as the Knative operator installation. However, this task installs cluster-wide resources that should not be handled by project-specific infrastructure code anyway.

Switching the deployment from Minikube to OKD was straightforward, as the infrastructure definitions are mostly platform-agnostic. Using different contexts for different clusters made it easy to switch between environments.

All in all, OpenTofu significantly improved our workflow and will be a valuable tool for future projects.

\subsection{Preparing a cluster}

As stated before, we had to switch from GCP to a private OKD cluster due to account issues. This change required us to adapt our implementation plan and learn how to work with Kubernetes-based solutions. Then, waiting for the cluster administrator to install the Knative operator caused significant delays.

Local development with Minikube allowed us to continue working on the project while waiting for the OKD cluster to be prepared. However, some difficulties with Minikube on Windows caused additional delays again.

Here, we learned the importance of having a well-prepared cluster environment - in the best case one that we can fully manage ourselves. It may have been better to switch to a different public cloud provider instead of using a private cluster, as this would have allowed us to have full control over the environment and may have avoided the trouble with the Knative operator installation.

\subsection{Minikube as Stand-In}

Minikube served as a valuable stand-in for the OKD cluster during development. It allowed us to continue working on the implementation while waiting for the OKD cluster. However, Minikube and other docker-based Kubernetes solutions come with their own set of challenges, especially on Windows systems.

Especially networking issues as well as issues with virtualization were encountered, which caused some frustration and debugging sessions.

While local solution such as Minikube sound great for such use cases, it is important to be aware of the potential pitfalls. In our experience, it may be a good tool for prototyping and short-term development while waiting fow a more permanent solution, but it should not be relied upon for long-term development.
